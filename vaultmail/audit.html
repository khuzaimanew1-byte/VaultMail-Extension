3 Preview Tab Audit
What powers the Preview tab?
Primary state keypreviewStatus
Secondary keycapturedEmail
Tertiary keycurrentFieldEmail (read but not currently rendered)
Display-only keydetectedSelectors
QuestionAnswer
Where does Preview get data from? chrome.storage.local on popup open via loadState(); live updates via chrome.runtime.onMessage and chrome.storage.onChanged
Is Preview persistent? PARTIAL — previewStatus is persistent but reset to 'activated' on every Replit page load/refresh. capturedEmail persists across restarts.
When does Preview update? On messages: CONTEXT_LOCKED, CURRENT_EMAIL_CHANGED, EMAIL_CAPTURED, CONTEXT_RESET, REPLIT_STATUS_CHANGED. Also on any storage change for its keys.
When does Preview clear? Returns to Activated state on: SPA navigation, page refresh, tab close, context reset (no valid form+submit found)
Can Preview survive browser restart? PARTIAL — Popup will reopen to Activated state (content script not running). capturedEmail is still in storage and will show in Previewing Account if user navigates back to Replit.
Can Preview survive extension reload? PARTIAL — Same as restart. Storage persists; live state is lost until content script re-runs on the page.
4 Emails Tab Audit
QuestionAnswer
Which storage key provides emails? vaultmail_emails — string array in chrome.storage.local
How are emails added? User pastes/types into Add modal textarea → parseEmails() splits on newlines/commas, validates format via regex, lowercases all → merged with existing array using Set() deduplication → saveEmails() writes to storage
How are duplicates handled? [...new Set([...emails, ...parsed])] — deduped at write time. Cross-session dedup works because existing emails are loaded first. Case-insensitive: all lowercased before comparison.
How are emails deleted? Delete button → openDeleteModal(email) → user confirms → confirmDelete() → emails.filter(e => e !== target) → saveEmails(). Card plays CSS removal animation first.
What survives restart? ALL emails — vaultmail_emails is in chrome.storage.local, persists indefinitely
What survives tab refresh? ALL emails — tab refresh does not affect chrome.storage.local

9 Orphaned Data Detection
🔴 background.js — Storage Key Mismatch

background.js writes keys that popup.js never reads. These silently pollute storage.

bg.js writespopup.js readsStatus
captureStatuspreviewStatusMISMATCH — orphan write
currentActiveEmailcapturedEmailMISMATCH — orphan write
background.js was written against an older popup.js API. The current popup.js was refactored to use content.js direct messages, but background.js was not updated to match.
Recommendation: Remove captureStatus and currentActiveEmail writes from background.js, or update them to match current key names previewStatus / capturedEmail.
🔴 background.js — Message Type Mismatch
bg.js broadcastspopup.js handlesStatus
STATUS_CHANGEDNot listed in onMessage switchORPHAN — never handled
ACTIVE_EMAIL_UPDATEDNot listed in onMessage switchORPHAN — never handled
popup.js handles: CONTEXT_LOCKED, CURRENT_EMAIL_CHANGED, EMAIL_CAPTURED, CONTEXT_RESET, REPLIT_STATUS_CHANGED, SELECTORS_UPDATED. None of these are sent by background.js.
Recommendation: background.js should send REPLIT_STATUS_CHANGED (with active: bool) which popup.js already handles. Remove STATUS_CHANGED and ACTIVE_EMAIL_UPDATED broadcasts.
🔴 replitActive — Undeclared Variable in popup.js
popup.js references replitActive in storage.onChanged listener and in the REPLIT_STATUS_CHANGED message handler, but let replitActive is never declared in the file's state block. This causes a ReferenceError or implicit global in non-strict contexts. Since popup.js uses 'use strict', this will throw at runtime.
Recommendation: Add let replitActive = false; to the state declarations at the top of popup.js. Also add 'replitActive' to the loadState() chrome.storage.local.get() call.
🟡 currentFieldEmail — Written to Storage, Not Rendered
currentFieldEmail is written to storage on every keystroke (content.js → onFieldInput) and read by popup.js loadState/onChanged, but it is not currently rendered anywhere in the popup UI. The popup.js CURRENT_EMAIL_CHANGED handler updates the in-memory variable but only calls renderPreview() conditionally — and renderPreview() does not output currentFieldEmail anywhere in the DOM.
Recommendation: Either render currentFieldEmail in the Processing state description, or stop writing it to storage to reduce churn (it updates on every keystroke).
🟡 ensureContextValid() — Dead Code Path
ensureContextValid() existed to handle DOM re-renders (React removing and re-adding submit buttons). It called attachSubmitListeners() — a function that no longer exists in the refactored codebase. The function was also called from makeSubmitHandler(). After the interaction-first refactor, this re-resolution path is partially dead. The attachSubmitListeners reference is undefined.
Recommendation: Remove ensureContextValid() or refactor it to only check DOM presence without re-attaching listeners.

QuestionAnswer
What does the extension capture?Email value from the locked input field at submit time (capturedEmail). Live typing value (currentFieldEmail). Page element presence (detectedSelectors). Manually-added emails (vaultmail_emails).
What does it remember across restarts?vaultmail_emails (all saved emails) and capturedEmail (last submitted email) survive all restarts, reloads, refreshes.
What does it forget?previewStatus (reset to 'activated' on every page load), currentFieldEmail (reset to null), in-memory context (activeEmailInput, activeForm, activeSubmitBtn, contextGeneration).
When does it forget?On Replit page load/refresh, SPA navigation, context reset (missing form or submit button).
What survives all restarts?vaultmail_emails, capturedEmail.
What never touches storage?In-content: activeEmailInput, activeForm, activeSubmitBtn, contextLocked, contextGeneration, scanThrottle. In-popup: activeTab, pendingDelete, toastTimer. In-background: tabUrls Map.
