Fix the email detection regression introduced by the recent status-system changes.

Current issue:

After the UI simplification, VaultMail remains stuck on "Extension Activated".

Even when:

* a valid email field is clicked
* a valid email field is focused
* login/signup forms are visible

the status never changes to Processing or Previewing Account.

This indicates the email interaction detection logic is either not firing, not matching the input correctly, or the status transition is broken.

Required tasks:

1. Fix email interaction detection.

The following must trigger Processing:

* focusin on a valid email field
* click on a valid email field
* direct user interaction with a valid email field

Supported email field patterns:

* input[autocomplete="email"]
* input[name="email"]
* input[name="username"][autocomplete="email"]

Do not rely on dynamic IDs.

2. Add a temporary Testing Panel below the status card.

Purpose:
Allow verification that interaction detection is working.

Show:

Clicked Element:

* tag name
* input/button/etc.

Detected Target:

* YES / NO

Email Candidate:

* YES / NO

Form Found:

* YES / NO

Submit Button Found:

* YES / NO

Current Status:

* Extension Activated
* Processing
* Previewing Account

Active Email Selector:

* detected selector used internally

This testing panel is temporary and only for debugging.

3. Add Detection Logs

When interaction occurs, log:

Event Received
Element Matched
Email Candidate Matched
Form Found
Submit Found
Context Locked

This will reveal exactly where the chain is failing.

4. Verify Status Flow

Expected flow:

Extension Activated
→ click/focus valid email input
→ Processing
→ context built successfully
→ Previewing Account

If any step fails, the Testing Panel must show which condition failed.

5. Architecture Verification

Audit the current implementation and explicitly report:

* Is the interaction-first architecture actually active?
* Is the extension still using any page scanning?
* Is any MutationObserver active?
* Is any interval/timer-based scanning active?
* Is any selector discovery loop still running?

Show these results inside the Testing Panel:

Interaction Listener:
ACTIVE / INACTIVE

Continuous DOM Scan:
YES / NO

MutationObserver:
YES / NO

Interval Scanner:
YES / NO

Context Locking:
ACTIVE / INACTIVE

6. Performance Requirement

Do not add new scanning systems.

The goal is to identify why interaction detection is failing, not to reintroduce DOM-wide scanning.

The testing panel should help verify that:

* user interaction is being received
* email candidate detection works
* form detection works
* submit button detection works
* context locking works

without reintroducing expensive monitoring logic.

Once the issue is fixed and verified, the testing panel can be removed later.
