## INITIALIZATION

Fix context initialization.

Whenever VaultMail starts, reconnects, or becomes active, it must correctly establish its initial state from the current page context without requiring any manual page refresh or user workaround.

This must work for all equivalent situations, not just a specific page-load scenario.

---

## STATUS SYSTEM

Only 4 states exist.

State 0:

Idle

Description:
The extension is no longer needed

Trigger:

* No usable website context exists
* All Replit tabs are closed

---

State 1:

Extension Activated

Description:
Waiting for email interaction

Trigger:

* Website context exists
* No active email context

---

State 2:

Processing

Description:
Email detected. Preparing capture.

Trigger:

* User interacts with a valid email field
* New email context is being established

During Processing:

1. Set activeEmailInput.
2. Resolve the parent form.
3. Find the submit button inside the same form.
4. Lock the context.

State 3:

Previewing Account

Description:
Captured email

The description must display the captured email itself.

Example:

Previewing Account

[mailing5@mailbox.org](mailto:mailing5@mailbox.org)

Trigger:

* Submit button clicked
* Email captured successfully

---

## CONTEXT SWITCHING

When the user interacts with a different valid email field:

1. Completely discard the previous active context.
2. Create a new context from the newly interacted email field.
3. Resolve its parent form.
4. Resolve its submit button.
5. Change status to Processing.

The previous context must never remain active after a new email context is selected.

---

## PREVIEWING STATE

The captured email must remain displayed until another email is successfully captured.

only in idle state the captured email shows instead of its original description, if this state comes during previewing
